# 《Vibe Coding 游戏开发：需求完备度、测试体系与 Subagent 协作法》

## 0. 文档目的

本文回答三个实战问题：

1. 一款游戏在开始开发前，至少要明确多少类信息，Agent 才不会迷茫。
2. 为什么本项目里要拆出二十多个小测试，它们各自解决什么问题。
3. Subagent 是否真的能提高效率，尤其是在 debug、策划审阅、迭代收敛上应该怎么用。

本文不是泛泛而谈，而是基于 `ZJU Cat Merge` 这类移动端 2D Web 游戏的实际开发经验总结。

---

## 1. 要让 Agent 不迷茫，至少要明确哪些信息

结论先说：  
如果是一个中小型 2D Web 游戏，真正能让 Agent“稳定开工”的，不是 5 条需求，也不是 10 条需求，而是至少 `25-40 个明确决策项`。  
如果想减少返工、降低 hallucination、减少后续追问，建议直接准备 `40+` 项信息。

下面给出一份适合游戏项目启动的完整清单。

### 1.1 产品定位层

1. 项目代号 / 正式名称
2. 目标玩家是谁
3. 核心场景是什么
4. 这款游戏最想传递的情绪是什么
5. 玩家为什么会愿意玩第二局
6. 玩家为什么会愿意分享
7. 首发版本是验证玩法、验证传播，还是验证付费
8. 成功标准是什么

### 1.2 发行与平台层

9. 首发载体是 H5、普通 Web、微信内 H5、还是小程序
10. 是否手机优先
11. 是否桌面兼容
12. 是否竖屏固定
13. 是否允许横屏
14. 是否要求微信内直接可玩
15. 部署方式是什么
16. 是否需要联网

### 1.3 技术架构层

17. 前端渲染主栈是什么
18. 物理引擎是否存在
19. UI 是 Phaser 内绘制，还是 DOM overlay
20. 前后端是否分离
21. 是否采用 Serverless
22. 数据库存什么
23. 是否需要匿名 ID
24. 是否有本地存档
25. 是否需要事件总线或共享状态层

### 1.4 核心玩法层

26. 核心循环是什么
27. 单次输入是什么
28. 输入反馈是什么
29. 掉落规则是什么
30. 碰撞规则是什么
31. 合成规则是什么
32. 分数规则是什么
33. Game Over 条件是什么
34. 是否存在宽容失败机制
35. 单局时长目标是多少
36. 难度曲线怎么走

### 1.5 生成与成长层

37. 掉落等级是否固定
38. 直出等级上限是多少
39. 哪些等级必须靠合成获得
40. 掉落概率分布是什么
41. 是否与分数挂钩
42. 同等级是否有多个外观 variant
43. 是否要防止连续刷到同一只
44. 合成上限是多少
45. 最终目标实体是什么

### 1.6 道具与系统层

46. 是否有道具
47. 道具有哪些
48. 每个道具如何触发
49. 是即时生效还是指向生效
50. 道具获取方式是什么
51. 每局次数是多少
52. 道具是否影响主循环 CD
53. 道具是否影响 Combo
54. 道具是否触发红线豁免

### 1.7 爽感与反馈层

55. 合成反馈做几层
56. 是否有 Q 弹、闪光、震屏、浮字
57. 哪些反馈只作用于视觉层
58. 哪些事件算高潮事件
59. Combo 时间窗是多少
60. Combo 只是显示还是参与加分
61. 高等级合成是否有特殊反馈
62. 失败时的反馈要多强

### 1.8 美术与资源层

63. 美术主题是什么
64. 配色体系是什么
65. 字体方向是什么
66. 资源是 PNG、GIF、Spine 还是程序生成
67. 原始素材是否需要预处理
68. 运行时是否允许裁切
69. 透明底图片是否直接使用
70. 资源命名规则是什么
71. 是否需要为未来动态资源预留接口

### 1.9 UI / UX 层

72. 首页是否单独存在
73. 游戏内 HUD 包含什么
74. 结算页包含什么
75. 排行榜入口在哪里
76. 设置入口在哪里
77. 改昵称入口在哪里
78. Game Over 后主按钮是什么
79. 工具按钮布局如何
80. 哪些内容必须首屏可见
81. 是否允许页面滚动

### 1.10 联网与运营层

82. 是否有排行榜
83. 是总榜、周榜还是双榜
84. 排行榜字段有哪些
85. 是否有分享页
86. 是否以截图分享为主
87. 是否做防作弊
88. 防作弊强度到哪一级
89. 是否需要昵称审核
90. 是否需要埋点

### 1.11 音频层

91. 是否有音效
92. 是否有 BGM
93. 音乐是否可商用
94. 是否有音效开关
95. 是否有音乐开关
96. 是否有震动反馈开关

### 1.12 开发边界层

97. V1 必做什么
98. V1 明确不做什么
99. 哪些内容允许先 mock
100. 哪些内容绝不能先糊一个版本
101. 验收顺序是什么
102. 哪些问题优先级最高

### 1.13 真正足够开工的标准

不是每项都要 100% 细化，但如果下面这些还没定，Agent 通常会明显迷茫：

- 平台与载体
- 技术主栈
- 核心玩法循环
- 输入模型
- 失败规则
- 等级与生成规则
- 资源形态
- UI 主结构
- V1 边界
- 成功标准

如果以上 10 类都清楚，Agent 可以开工。  
如果以上 10 类之外再补上 20-30 个参数，Agent 就不只是“能开工”，而是“能稳定推进、少返工”。

## 2. 为什么本项目要拆出二十多个测试

### 2.1 先说结论

本项目当前测试文件数是 `21` 个。  
这不是为了“显得专业”，而是为了把高风险逻辑拆开钉死，避免每修一个点就把另一个点修坏。

2D Web 游戏的 bug 很少是单点错误。大多数 bug 都是：

- 物理 + 输入 + 状态 + UI 的耦合错误
- 同一条逻辑在不同帧、不同层被重复消费
- 一个“看起来只是 UI 的问题”，实际根因在状态机或输入链

所以测试策略不能只写 1 个“大而全”的集成测试，而是要按“风险边界”拆成很多小测试。

### 2.2 本项目测试的分层逻辑

#### A. 启动与壳层 smoke test

- `tests/smoke/app-start.test.ts`
- `tests/smoke/mobile-shell.test.ts`

作用：

- 确认应用能启动
- 核对首页、游戏壳、移动端基础结构没有直接炸掉

适用场景：

- 大改首页
- 大改挂载逻辑
- 大改 DOM HUD

#### B. 纯规则测试

- `tests/game/merge-system.test.ts`
- `tests/game/danger-system.test.ts`
- `tests/game/run-state.test.ts`
- `tests/game/run-state-reroll.test.ts`
- `tests/game/tool-state.test.ts`

作用：

- 验证“规则是否成立”
- 与渲染无关
- 与 Phaser 实例生命周期无关

这些测试最重要，因为它们是“玩法真相”的最后防线。

#### C. 物理与空间策略测试

- `tests/game/physics-policy.test.ts`
- `tests/game/merged-spawn-placement.test.ts`
- `tests/game/playfield-layout.test.ts`
- `tests/game/cat-sizing.test.ts`

作用：

- 确认球的出生点、大小曲线、场地布局、物理策略不会把玩法直接搞坏

这些测试的价值在于：  
它们能提前拦住“看起来只是调参数，实际上会导致卡死”的修改。

#### D. 合成流程测试

- `tests/game/merge-queue.test.ts`
- `tests/game/merge-feedback-policy.test.ts`
- `tests/game/merge-animation-policy.test.ts`

作用：

- 钉死“合成时不能直接在碰撞回调里做完整事务”
- 钉死“合成反馈不能作用于物理体”
- 钉死“合成队列必须走安全时机”

这些测试本质上是在防止“又回到会卡死的老路”。

#### E. 输入链测试

- `tests/game/input-policy.test.ts`
- `tests/game/hud-action-queue.test.ts`
- `tests/game/hud-bridge.test.ts`
- `tests/game/dom-hud.test.ts`

作用：

- 处理 DOM HUD 和 Phaser Scene 双层输入时最容易炸的地方
- 验证按钮点击不会污染下一次按住生成
- 验证 HUD 命令不是在错误时机直接改 Scene

这些测试在 DOM overlay 改造之后尤其关键。

#### F. 特殊玩法测试

- `tests/game/bomb-effect.test.ts`
- `tests/test_process_cat_assets.py`

作用：

- `Bomb` 这类复杂范围道具需要单独钉效果边界
- 资源预处理脚本需要防止导出结果悄悄失真

### 2.3 为什么是“小测试很多”，而不是“大测试很少”

因为小测试有 4 个现实优势：

1. 改一个点，只跑相关测试，速度快  
2. 失败时能直接定位责任层  
3. 更适合 Vibe Coding 这种高迭代、频繁重构的节奏  
4. 更适合防止“修好 A，顺手弄坏 B”

### 2.4 后续项目的测试建议

一个类似规模的 Web 游戏，建议最少从这 8 类测试起步：

1. 启动 smoke
2. 核心规则
3. 失败机制
4. 生成逻辑
5. 合成流程
6. 输入链
7. 物理/布局策略
8. 关键道具

能做到这 8 类，Agent 的 debug 成本会下降非常明显。

## 3. Subagent 是否真的能提高效率

结论：  
可以，而且在复杂游戏项目里很有价值。  
但前提是：`Subagent 只能做边界清晰的 sidecar 工作，不能拿来替代主线架构判断。`

### 3.1 最适合给 Subagent 的任务

1. 审查测试输出并提出修复方向
2. 只看某一条输入链是否合理
3. 只看某个 UI 结构是否满足设计目标
4. 只做一块文档整理
5. 只验证某个系统的风险点
6. 只负责一个明确文件集的小改动

这类任务的共同点是：

- 边界明确
- 输出格式明确
- 不直接决定整体架构

### 3.2 最不适合给 Subagent 的任务

1. 替你决定整个项目方向
2. 在没有全局上下文时乱改主循环
3. 直接接管一个跨 5 个系统的大重构
4. 在主线 blocker 上“先扔给别的 agent 看看”

原因很简单：  
主线架构、状态机边界、系统耦合判断，必须由主 agent 统一收口。

### 3.3 你提出的工作流是否可行

你的想法是：

- 我先测试
- 我输出当前结果、问题和下一步计划
- 你把这些作为输入发给一个“资深游戏策划师”或其他 subagent
- 让它给修改建议
- 再回到主 agent 继续实现

这个工作流是可行的，而且是高价值用法。

但要注意 3 条约束：

#### 约束 1：Subagent 应该审方案，不直接拍代码

比如可以让它回答：

- 当前手感问题更像数值问题还是结构问题
- 这轮改动是否有设计回归
- 这条失败机制是否符合“轻挫败”目标
- 这套 UI 是否违背产品情绪

这是非常适合“资深游戏策划师 subagent”的。

#### 约束 2：输入必须是结构化的

不要只扔一句“帮我看看哪里不对”。  
最好提供：

- 当前版本表现
- 已知问题
- 已尝试方案
- 不可违背的设计约束
- 主 agent 下一步计划

这样 subagent 才能输出高质量建议。

#### 约束 3：Subagent 输出是建议，不是最终裁决

主 agent 必须做最后的技术收口：

- 哪些建议可落地
- 哪些建议和当前架构冲突
- 哪些建议会制造新耦合

### 3.4 一个推荐的 Subagent 协作 SOP

适合游戏开发的循环是：

1. 主 agent 先定义当前问题边界
2. 主 agent 给出已知根因与已排除项
3. Subagent 只针对该问题给建议
4. 主 agent 决定是否采纳
5. 主 agent 负责落代码、跑测试、整体验证

这是一种“主 agent 负责闭环，subagent 负责增量判断”的模式。

### 3.5 什么时候开 Subagent 最值

- 需求模糊，需要一轮设计批判
- debug 卡在“有多个可能根因”
- UI 已可用，但需要风格收敛建议
- 主 agent 需要并行处理两个互不相干的问题
- 需要独立审文档、审测试、审交互

### 3.6 什么时候不开更好

- 已经知道根因，只差动手修
- 当前是紧急 blocker
- 任务边界不清
- 主 agent 还没建立基本上下文

## 4. 对未来项目的推荐工作法

### 4.1 启动阶段

先把需求问到至少 25-40 项，再让 Agent 正式开工。  
不要让 Agent 在“平台没定、玩法没定、失败规则没定、资源没定”的情况下直接写代码。

### 4.2 开发阶段

优先建设：

1. 核心规则测试
2. 关键输入链测试
3. 物理/空间策略测试

然后再堆：

4. UI
5. 道具
6. 联网
7. 分享

### 4.3 Debug 阶段

遇到 bug 时必须先问三件事：

1. 根因更像物理、状态、输入，还是 UI 映射
2. 是否有独立测试能先把它钉住
3. 当前修复是在补丁，还是在恢复正确边界

### 4.4 协作阶段

主 agent 做：

- 架构
- 收口
- 测试
- 集成

subagent 做：

- 审阅
- 并行分析
- 局部建议
- 限定范围的实现

---

## 5. 最终结论

一款 Agent 能高效开发的游戏，不是“需求大概讲清楚”就够了，而是要把：

- 产品目标
- 核心玩法
- 失败规则
- 技术主栈
- 资产策略
- UI 结构
- 测试边界
- 协作方式

都明确到可以拆成独立责任层的程度。

真正让 Vibe Coding 成功的，不是“AI 很强”，而是：

- 需求足够成体系
- 测试足够成边界
- 主 agent 与 subagent 的职责足够清楚

这三件事一旦成立，游戏开发就会从“不断救火”转向“稳定推进”。

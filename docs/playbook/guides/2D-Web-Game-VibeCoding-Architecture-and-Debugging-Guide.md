# 《2D Web 游戏 VibeCoding 架构与排错指南》

## 0. 文档定位

本文不是项目总结，而是后续所有 2D Web 游戏项目的工程母法。

它沉淀了本项目从立项、系统设计、玩法打磨、移动端适配，到多轮重构、输入链修复、UI 迁移、编码污染处理的完整经验。目标不是记录“做了什么”，而是明确：

- 哪些架构决策值得保留
- 哪些错误绝不能再犯
- 哪些现象看似随机，实则是边界失控
- 后续任何 AI 协作开发应遵守哪些刚性约束

本文默认技术背景为：

- 渲染与场景：`Phaser`
- 物理：`Matter.js`
- 外层界面：`DOM / CSS`
- 平台：`移动端优先的 Web 游戏`

## 1. 引擎与物理架构层

### 1.1 本项目最棘手的物理/渲染解耦问题

本项目最重的技术坑，不是单一 bug，而是“把视觉资源直接当物理实体”的连锁灾难。

我们前期多次出现以下问题：

- 合成瞬间炸开
- 大球生成后卡死
- 球停在半空，像死锁
- 堆叠不稳定，盘面越往后越脆
- 透明 PNG 轮廓与实际碰撞边界完全不一致
- 合成反馈或缩放动画一加，物理体就失真

根因最终收敛为三条：

1. 透明猫图的视觉形状极不规则，但物理系统需要稳定的几何体
2. 在物理帧中直接 `destroy + spawn + tween scale`，会把 Matter 的碰撞解算打断
3. 把“视觉反馈”做在物理体本身上，而不是独立视觉层

最终可用解法是：

- 物理层永远使用稳定圆形刚体
- 猫图只是挂在刚体上的表现层 sprite
- 合成反馈不再缩放物理体，只做独立视觉反馈
- 合成球的出生点必须考虑新半径，不能简单取旧球中点

这一步是整局从“能跑”到“可信”的分水岭。

### 1.2 Phaser + Matter.js 核心开发铁律

#### 铁律 1：物理体必须比美术更简单

透明 PNG、GIF 首帧、表情包素材都只能服务表现，不能反向决定碰撞体。

结论：

- 用圆形刚体托住玩法稳定性
- 用图片层托住风格表达
- 绝不为了“更像素材轮廓”而牺牲物理一致性

#### 铁律 2：绝不在碰撞回调里同步完成整套合成事务

`collisionStart` 中只能做判定和入队，不能做：

- 销毁旧球
- 生成新球
- 播完整反馈
- 连续触发更多状态变更

正确做法：

- 碰撞帧内只打锁、入队
- 在安全时机统一处理合成队列

这条规则直接决定是否会出现重复合成、重复加分、场景卡死。

#### 铁律 3：不要对 Matter body 做 scale / tween 形变

物理体的缩放、回弹、视觉膨胀，都会引入碰撞边界漂移与解算异常。

正确做法：

- 物理体尺寸固定
- 爽感动画只作用于视觉层、光圈层、浮字层

#### 铁律 4：合成新球出生位置必须是“几何安全点”

不能只用两球中心点生成新球，尤其在中高等级时，新球半径更大，中心点往往已经与地面或邻球重叠。

正确做法：

- 根据 `mergedRadius` 重新计算出生点
- 将新球抬到不会立即穿插的位置

#### 铁律 5：表现复杂时，优先做减法，不要反向加复杂碰撞

本项目明确证明：运行时复杂裁切、动态底板重绘、非规则边界拟合，都属于高风险低收益。

## 2. 状态机与游戏逻辑层

### 2.1 高频事件与重复处理的核心坑

我们在连续碰撞、连锁合成、道具输入、重开一局等高频逻辑中反复踩过同一类坑：事件边界不清，导致状态重复消费。

最典型表现：

- 同一对球被判定多次合成
- `Restart` 后残留旧局状态
- 道具点击后把下一次输入链一起污染
- DOM HUD 与 Phaser 场景双向通信后，按钮看似响应，实际把 Pointer 状态弄脏

这些问题本质上都是“状态变更没有单一入口”。

### 2.2 稳定控制流的正确写法

#### 原则 1：所有高风险状态变更都要有单一消费点

例如：

- 合成：统一走 merge queue
- HUD 命令：统一在场景安全时机消费
- Game Over：统一由危险状态机给出
- 重开：统一 reset 整局，不允许局部手工清理

#### 原则 2：输入来源必须分层

本项目后期最大的问题之一，是把 `DOM HUD` 事件和 `Phaser Scene` 事件直接揉在一起。

必须明确：

- DOM 按钮只负责“发命令”
- 场景只负责“消费命令”
- 场内点选仍由 Phaser 自己处理

否则很容易出现：

- 点按钮后 Pointer 状态错乱
- 目标道具激活后无法选球
- 下一次按住生成球失效

#### 原则 3：任何“重开一局”都必须是完整重置，而不是局部清空

需要统一重置：

- 盘面球体
- 分数
- 危险值
- 道具次数
- `queued next`
- merge queue
- active tool
- 目标选择态
- HUD 可见状态
- Pointer 抑制状态

只清一半，之后的问题都会以“偶发”形式出现。

#### 原则 4：Phaser 与第三方 DOM 交互时，要显式处理 Pointer 生命周期

这是本项目后期非常关键的一条教训。

DOM 覆盖层会抢走输入事件，导致 Phaser 侧认为某个 Pointer 仍处于按下、未释放、或状态不一致。

必须使用：

- `this.input.resetPointers()`
- `pointerupoutside`
- 明确的输入抑制清理逻辑

否则所有“偶发不能拖动”“点按钮后假死”“目标道具不能选球”都会集中爆发。

### 2.3 防死机、防状态爆炸的核心约束

- 任何跨层命令，不要在外层事件回调里直接重写场景状态
- 高频路径中，不允许既改状态又立刻触发更多事件分支
- 所有布尔状态都要问一句：谁设置，谁清理，何时清理
- 如果某个状态的清理时机说不清，这个状态设计本身就有问题

## 3. 游戏设计与玩家体验层

### 3.1 红线失败机制的演化

本项目在失败机制上的最大正确决策，是尽早放弃“碰线即死”。

最终稳定方案是：

- 红线只针对“落稳危险体”
- 越线后累计危险值
- 达到阈值才 Game Over
- 回落或消除后快速衰减
- 道具触发时给短暂豁免

这套机制的价值在于：

- 保留压迫感
- 避免物理抖动误杀
- 给玩家自救空间

这就是“轻挫败”的真正实现方式：不是削弱失败，而是让失败看起来公平。

### 3.2 道具系统的设计教训

四个道具最终成立，是因为它们被拉回了“功能清晰、触发明确、系统保护充足”的路径：

- `Refresh`：改 `Next`
- `Shake`：扰盘
- `Hammer`：单点精准删除
- `Bomb`：范围删除 + 外圈击飞

真正重要的不是道具种类，而是这三条约束：

1. 破坏性道具必须有红线豁免
2. 目标道具必须有明确的激活态与取消态
3. 道具不能污染正常掉落节奏

### 3.3 “轻挫败”的实现原则

- 失败必须来自玩家可理解的局势，而不是系统噪声
- 玩家要感觉“再来一把我能更好”，而不是“系统刚才阴我”
- 自救手段要足够少，避免完全无脑；也要足够强，避免形同虚设

### 3.4 “爽感反馈”的实现原则

本项目一个关键结论是：爽感不是靠把每个反馈都做重，而是靠“正确分层”。

正确分层：

- 普通碰撞：轻
- 有效合成：明确
- 连续合成：递进
- 高价值事件：高潮

错误做法：

- 全程都很吵
- 所有反馈同一个强度
- 用物理爆炸代替情绪高潮

最终结论：

- 爽感来自节奏设计，不来自粗暴抖屏
- 高潮必须稀缺
- 视觉反馈绝不能破坏读局

## 4. 资产预处理与 UI 适配

### 4.1 素材处理的教训

本项目已经证明，透明底图的正确用法是“预处理”，不是“运行时智能裁切”。

我们后期确定的资产原则是：

- 原始 GIF 只存档
- 游戏只吃发布态 PNG
- 统一透明底
- 统一正方形画布
- 统一缩放规则

不要在运行时做：

- 复杂裁切
- 底板重绘
- 临时算包围盒
- 非统一尺寸推断

因为这些工作都应该在资产管线解决，而不是在游戏帧循环里解决。

### 4.2 移动端全屏适配的教训

我们前期多个问题都来自“把游戏放进一个还会滚动的网页卡片里”。

最终正确方案是：

- 首页和游戏态分离
- 进入游戏后全屏接管
- 页面禁滚动
- 游戏区占满横向空间
- HUD 独立覆盖，不嵌在游戏 canvas 内部

这条经验非常关键：移动端游戏不是普通网页组件。只要还在和外层滚动容器抢手势，玩法就不可能稳定。

### 4.3 做减法的渲染原则

任何下面这些“看起来更高级”的方案，只要没有明确收益，都应该删：

- 多层边框
- 调试感分隔线
- 运行时花式裁切
- 不必要的视觉组件堆叠
- 难以维护的动态 UI 组合

本项目后期视觉真正变好的原因，不是加了更多东西，而是删掉了：

- 多余边框
- 多余灰条
- 硬分区
- 过度复杂的场内 HUD

最终结论：

> Web 游戏 UI 的高级感，来自统一材质和层次，不来自元素数量。

## 5. AI VibeCoding 协作反思

### 5.1 本次协作中 AI 犯过的关键错误

#### 错误 1：过度打补丁，而不是承认架构已经错了

最典型的阶段是：

- 合成炸开
- 大球卡死
- DOM HUD 按钮假死

前几轮都存在“再补一个 if、再调一个参数、再加一层 suppress”的倾向。
这种做法短期看像在推进，长期看是在堆脆弱性。

正确做法应是：

- 先停下
- 承认当前结构不可信
- 回到更小、更稳定的边界重新切层

#### 错误 2：没有先确认全局耦合范围，就修改输入链

本项目后期最危险的一类错误，就是为修一个按钮问题，连续改了：

- DOM 事件模型
- bridge
- queue
- suppression
- Phaser Pointer 状态

如果修改前没有明确回答“这条链影响哪些系统”，基本必然引入次生问题。

#### 错误 3：把视觉问题和功能问题绑在一起改

这是典型的不专业行为。

例如用户只是要求改 UI，但如果在同一轮顺手重写了执行入口、输入状态机、按钮事件时机，最后就会出现：

- UI 变好了
- 功能一起坏了

这类问题完全是过程纪律问题，不是技术难度问题。

#### 错误 4：忽略编码链路污染风险

我们后期出现的多处 `????`、乱码、错误替换，就是典型的“在已有污染链上继续局部字符串替换”。

正确做法应是：

- 一旦确认编码链不可信，整段重写
- 优先使用 ASCII 或 `\u` 转义
- 不在污染文件上做逐词修补

#### 错误 5：在未确认根因前重复扩大搜索和修改面

这类行为会带来两种后果：

- 用户体感是“越修越乱”
- AI 自己丢失原始可用路径

### 5.2 未来 AI 协作的绝对准则

#### 准则 1：重构优于打补丁

如果一个问题已经涉及：

- 输入链
- 场景生命周期
- 物理架构
- 跨层通信

那么优先考虑重构边界，而不是继续在旧边界上加条件。

#### 准则 2：修改前必须先确认影响范围

每次动代码前，必须先明确：

- 这条逻辑被谁调用
- 会影响哪条用户手势链
- 是否触发连锁状态变化
- 是否跨越了 DOM / Phaser / physics / UI 多层边界

#### 准则 3：UI 改动与功能改动默认拆开

除非用户明确要求，不允许把“重构视觉”和“改写功能入口”绑成一轮。

#### 准则 4：发现架构性问题时，要主动提出回退点

不是每次都向前硬顶。
成熟做法是明确告诉用户：

- 当前方案哪里不可信
- 最近的稳定回退点是什么
- 重建路径是什么

#### 准则 5：所有“偶发 bug”都按状态机污染处理

只要用户说：

- 有时失效
- 偶尔卡死
- 点完按钮下一次不能操作

优先怀疑：

- 状态未清
- 输入源错乱
- 生命周期时机错误

而不是先怀疑 UI 或素材。

### 5.3 未来项目的 AI 协作 SOP

1. 先界定问题层级：视觉、输入、状态、物理、通信
2. 一次只改一层
3. 先加最小测试，再动实现
4. 改完先跑针对性回归，再跑 smoke/build
5. 如果连续两轮补丁仍未稳定，强制进入“回退/重构模式”

## 6. 最终工程纲领

### 6.1 架构总原则

- 物理必须简单，表现可以丰富
- 场景必须单向消费高风险状态
- 输入来源必须分层
- 素材问题在资产管线解决，不在运行时硬算
- UI 高级感来自统一材质，不来自堆组件

### 6.2 排错总原则

- 偶发问题优先查状态污染
- 连锁问题优先查执行入口
- 合成问题优先查物理边界和事务时机
- 移动端问题优先查手势所有权
- 编码问题优先整段替换，不做局部赌运气修复

### 6.3 协作总原则

- 重构优于打补丁
- 修改前先画影响范围
- UI 与功能默认拆轮
- 任何跨层通信都要有清晰边界
- AI 不是“多写几行补丁”的工具，而是“先识别正确边界，再落实现”的工程助手

---

## 7. 一句话结论

本项目最重要的产出，不是一个可玩的猫咪合成游戏，而是一条已经被真实代价验证过的开发规律：

> 2D Web 游戏要稳定，必须先把物理、输入、状态、UI、资产五条边界切干净；任何一条边界偷懒，最后都会以“偶发 bug”和“越修越乱”的形式偿还。
